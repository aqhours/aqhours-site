# AQHOURS — Skyglass Design

## 1. 文档作用

本文件是 aqhours 新版个人站点的设计决策来源。当前阶段只讨论并确认设计，
不开始页面实现，不添加 3D 依赖，不制作正式组件。

后续实现必须以本文件为准。设计发生变化时，先更新本文档，再修改代码。

## 2. 当前阶段

### 已确认

- 旧版设计规范、旧版页面构图和旧版明暗主题不再作为约束。
- 首页首屏采用沉浸式天空场景。
- 首屏核心视觉是原创的透明玻璃字母骨架。
- 字母骨架固定采用完整 `Aqhours` 连字：首字母 `A` 大写，其余字母小写，并从 `A` 连续连接到结尾 `s`。
- 页面提供晨、昼、暮、夜四套完整场景光照。
- 场景光照不是传统网页浅色/深色主题。
- 首屏通过滚动完成空间变化和内容揭示。
- 整个站点保持个人表达，不采用产品官网式营销结构。
- 参考 Air 的空间关系与设计方法，不复制其字形、素材、文案或精确构图。

### 尚未确认

- 首屏可读文字与一句话介绍。
- 导航栏目名称。
- 实际展示的项目、照片、文章或其他个人内容。
- 是否按访客本地时间自动选择初始光照。
- 后半页各章节的最终数量与顺序。

在这些内容确认前，不允许使用占位文案冒充正式内容。

## 3. 设计愿景

aqhours 应该像一次进入天空空间的过程，而不是一张常规个人简历。

初次进入时，访客先看到云层、光与巨大透明字形。随着滚动，字形从主角变成空间
中的对象，aqhours 的身份和内容逐渐出现。后半页回到更安静的排版与真实材料，
让首屏的视觉冲击得到停顿，也让内容可以被认真阅读。

体验关键词：

- 明亮而有纵深
- 空气感
- 透明与折射
- 巨大尺度
- 安静的界面
- 克制的个人表达
- 连续的滚动叙事

## 4. 参考边界

可以参考：

- 全视口 WebGL 天空场景
- 巨型透明字形与云层的尺度关系
- 固定场景与滚动内容之间的衔接方式
- 多套环境光照对同一材质的整体影响
- 极简导航直接覆盖在场景上
- 压缩字体与大尺寸文字带来的节奏

不可以复制：

- 第三方品牌名称、标志和文案
- 原站字形轮廓或签名字体
- 原站 3D 模型、贴图、云层、HDRI、视频或截图
- 原站导航结构、产品卡片和营销 CTA
- 原站各屏幕的精确构图与滚动时间线

aqhours 必须拥有自己的字母骨架、材质调校、光照色彩和页面内容。

## 5. 整页叙事

整个页面采用五幕结构。首屏不是独立海报，而是第一幕和第二幕共同使用的空间。

| 幕 | 名称 | 作用 | 主要形式 |
| --- | --- | --- | --- |
| 1 | Arrival | 建立记忆点 | 天空、云层、玻璃字母骨架、光照控制 |
| 2 | Identity | 从视觉进入身份 | `Aqhours` 名称与一段极短的真实介绍 |
| 3 | Selected Work | 展示真实产出 | 少量大画面项目，不使用密集卡片墙 |
| 4 | Personal Index | 展示个人侧面 | 照片、文字或其他真实材料组成的轻量索引 |
| 5 | About / Finale | 完成介绍并留下出口 | 简短介绍、链接、联系方式、巨幅 AQHOURS |

页面应当有明显的张弛关系：

1. 第一幕视觉最强。
2. 第二幕开始出现信息。
3. 第三、四幕以内容为主，减少持续 3D 负担。
4. 第五幕重新用大排版形成结束感。

## 6. 全局界面

### 导航

- 导航固定在视口顶部，高度约 64–72px。
- 导航直接覆盖在场景或页面表面上，不使用常驻毛玻璃底板。
- 左侧为 `Aqhours` 标识或文字，右侧最多保留 2–4 个真实目的地。
- 不放置“注册”“预约”“开始使用”等产品式 CTA。
- 当背景明度变化时，导航颜色应平滑切换并保持可读。
- 移动端使用一个清晰的菜单入口，不把桌面导航强行压缩成一行。

### 页面表面

- 首屏由实时场景承担背景。
- 内容区主要使用纯黑、近黑或克制的浅色表面。
- 深度来自图像、空间、透明材质和表面对比，不依赖默认阴影。
- 玻璃材质只属于关键视觉对象，不用于所有卡片和导航。

## 7. 首屏：Skyglass Scene

### 7.1 空间结构

首屏的滚动容器建议占据约 `180–220vh`。WebGL 画布固定在视口内，HTML
导航和文字位于画布上方。

从后到前的视觉层级：

1. 天空颜色与大气雾
2. 远景和中景云层
3. 主玻璃字母骨架
4. 少量原创辅助形体
5. HTML 导航、光照控制与正文

辅助形体必须服务于主字形，不形成与参考站相同的球体和字形组合。

### 7.2 初始画面

- 第一眼不显示大段文字。
- 巨型玻璃字母骨架占据画面主体，并在视口边缘被自然裁切。
- 字形可超出视口宽度，但页面不得产生水平滚动。
- 云层必须具有前后距离，不是一张平铺背景。
- 导航保持可见，但视觉权重明显低于场景。
- 光照切换控件固定在右侧安全区域。
- 初始画面允许保留极小的 `Aqhours` 标识，但不放置营销按钮。

### 7.3 字母骨架

首屏采用完整的 `Aqhours` 玻璃连字，大小写固定为大写 `A` 加小写
`qhours`：

- 大写 `A` 是整组字形的视觉锚点，高度约为小写主体的 1.35–1.5 倍。
- 整组字形采用约 14–18° 的右倾连写姿态；倾斜应写入骨架本身，使每个字母的主笔共同右倾，不能只旋转整个单词假装斜体。
- `A` 必须保持可识别的左右主笔和横杠，不能为了连续曲线退化成类似小写字母的形状。
- `A` 必须直接接入 `q`，再由 `qhours` 连续延伸到结尾 `s`；不能把 `A` 做成独立字形后与后半部分并排放置。
- `A` 的横杠应由同一主笔回绕、交叉或穿插形成，不能使用悬空且不连接的短玻璃段。
- `qhours` 以较低、较流动的连续节奏从 `A` 向右展开。
- `q` 的下行笔画与 `h` 的上行笔画形成第一组纵向起伏，后面的 `ours` 逐渐放松。
- 每一个字母都必须与前后字母实际接触并连续，不允许出现视觉上分离的独立字母。
- 字母之间允许主笔回绕、自交和前后穿插，让一条连续骨架同时形成字腔和真实空间深度。
- 建模时可以为了拓扑和性能把网格分段，但所有可见分段必须无缝接触；最终视觉必须像一条从 `A` 贯穿到 `s` 的完整玻璃笔画。
- 管状截面不是完全规则圆形，应有轻微扁平和宽度变化。
- 初始构图中完整单词可以达到约 150–190vw，并在左右边缘被有意裁切。
- 滚动进入 Identity 前必须缩小并完整露出 `Aqhours`，让访客确认它是可读名称而不只是抽象形体。
- 字形必须从 `Aqhours` 自身的字母关系出发，不临摹参考站签名。
- 需要制作多版轮廓草图，再从正面识别度、侧面体积和滚动时轮廓三个角度选择。

正式制作模型前，必须先批准黑白轮廓和三个静态观察角度。

### 7.4 玻璃材质

玻璃应该像有厚度的透明实体，不是半透明白色管道。

必须具备：

- 清晰的外轮廓折射
- 内部多层反射
- 环境色进入玻璃内部
- 冰蓝或青色的轻微边缘色散
- 高光宽度随光照状态改变
- 足够的透明度，让云层可以穿过字形被看见
- 正面主体保持低覆盖率，背景应能明显穿透；字形清晰度主要来自外缘 Fresnel、环境反射和内部高光，不能依赖乳白色填充

避免：

- 过强霓虹描边
- 乳白塑料质感
- 镜面金属质感
- 大范围发光
- 为追求“玻璃”而牺牲字形轮廓

### 7.5 云层与天空

- 云层需要至少三个深度层级。
- 近景云可以轻微越过字形边缘，但不能长期遮挡主体。
- 云层运动极慢，主要用于维持空间生命感。
- 天空不得表现为简单 CSS 渐变；允许用渲染纹理、环境贴图或程序化大气。
- 四套光照共享同一空间与字形，只改变环境、曝光、雾色、云层明暗和玻璃反射。

## 8. 四套场景光照

场景使用“晨、昼、暮、夜”四个状态。它们是同一世界的四种时间，不是四个互不相关
的皮肤。

| 状态 | 天空与云层 | 玻璃表现 | 情绪 |
| --- | --- | --- | --- |
| 晨 | 淡紫、桃色、乳白，低角度柔光 | 暖色进入内部，边缘保持冰蓝 | 安静、刚刚醒来 |
| 昼 | 高饱和蓝天、明亮白云、高曝光 | 最清透，蓝白高光最强 | 开阔、自由、清醒 |
| 暮 | 薰衣草紫、淡金与柔雾 | 暖金反射与紫色边缘并存 | 浪漫、过渡、回望 |
| 夜 | 深海军蓝、低亮度云层、青色轮廓 | 更深、更厚，保留青色折射 | 私密、沉静、遥远 |

### 光照控制

- 桌面端使用右侧垂直四按钮。
- 移动端移动到右下安全区域，保证单手触达。
- 每个按钮必须拥有图标、可访问名称和明确的选中状态。
- 切换改变整个场景，而不是只替换背景色。
- 光照过渡允许使用 600–900ms 的连续插值，因为它是场景变化，不是高频 UI 响应。
- 按钮本身的按压反馈保持在 100–160ms。
- 用户切换后记住选择；是否首次按本地时间选择，仍待确认。

## 9. 首屏滚动时间线

滚动必须跟随用户输入，不劫持滚轮，不自动播放整段页面。

| 进度 | 场景变化 | 内容变化 |
| --- | --- | --- |
| 0–20% | 巨型字母骨架保持主导，仅有极轻微呼吸感 | 只显示导航和光照控制 |
| 20–45% | 字形开始旋转，部分笔画转为侧面，整体缓慢缩小 | `Aqhours` 标识逐渐出现 |
| 45–70% | 字形向上和远处移动，云层打开中央空间 | 一句短介绍和必要链接进入 |
| 70–90% | 字形退为背景对象，场景明度为正文让位 | Identity 内容成为视觉主角 |
| 90–100% | 场景平滑结束或停驻，准备释放渲染资源 | 页面进入 Selected Work |

运动要求：

- 字形旋转必须显示其真实厚度和前后穿插。
- 滚动映射需要平滑，但不能产生明显滞后。
- 不使用大幅鼠标跟随；若加入视差，最大旋转保持在约 2–3 度。
- 不允许元素从 `scale(0)` 出现。
- 不用持续弹跳或循环旋转制造“活泼”。
- 复杂动效只属于首屏叙事，普通导航和内容交互保持迅速。

## 10. Identity

Identity 与首屏共享天空场景，是从“看见 Aqhours”到“认识 Aqhours”的过渡。

- 使用完整、可读且大小写固定的 `Aqhours`。
- 最多一段很短的介绍。
- 文字内容必须由站点所有者确认。
- 不使用虚构口号、荣誉、经历或产品式价值主张。
- 不在同一画面加入多个按钮。
- 如果需要入口，优先使用朴素文字链接。

## 11. Selected Work

这一部分展示真实项目或作品，设计服从内容。

- 每个项目以一幅主视觉、项目名称和最少必要信息构成。
- 桌面端优先使用全宽或非对称双栏，不使用统一小卡片网格。
- 单屏只突出一个项目。
- 图片、视频和界面截图保持各自适合的比例。
- 项目之间使用大留白或黑色间隔形成节奏。
- 只展示真实存在并经过确认的项目。
- 没有足够内容时宁可减少项目数量，不制造占位项目。

## 12. Personal Index

这一幕用于展示项目之外的个人侧面，但具体内容仍待确认。

允许的形式：

- 有明确来源的照片序列
- 简短文字片段
- 文章或笔记入口
- 经过确认的兴趣、收藏或日常材料
- 少量混合比例的媒体网格

不允许：

- 为填满页面而生成虚构经历
- 仿社交媒体信息流
- 大量相同圆角卡片
- 没有叙事关系的图片墙

如果最终没有适合内容，这一幕可以删除。

## 13. About / Finale

### About

- 使用简短真实介绍和必要链接。
- 文字区域保持适合阅读的窄宽度。
- 联系方式清晰可访问，不使用模糊 CTA。

### Finale

- 回到纯黑或极深色画布。
- 使用巨幅压缩字体显示 `AQHOURS`。
- 字样可以触碰或超出视口边缘，但不得产生水平滚动。
- 保留必要链接、版权或联系方式。
- 结束画面安静，不再次加入复杂 3D 场景。

## 14. 排版

界面字体和显示字体必须使用可合法分发的字体。

| 角色 | 当前建议 | 用途 |
| --- | --- | --- |
| 界面与正文 | Inter + Noto Sans SC | 导航、正文、链接、标签 |
| 拉丁压缩显示 | Anton 或其他开源压缩字体 | 巨幅 AQHOURS、章节宣言 |
| 中文显示 | Noto Sans SC Black | 中文大标题 |
| 手写点缀 | LXGW WenKai / Caveat | 极少量亲密语气，不用于正文 |

建议字号：

| 角色 | 尺寸 | 行高 |
| --- | --- | --- |
| 巨幅显示 | `clamp(5rem, 18vw, 16rem)` | `0.82–0.88` |
| 首屏标题 | `clamp(2.75rem, 6vw, 4rem)` | `0.95–1.05` |
| 章节标题 | `clamp(2rem, 4vw, 3rem)` | `1.05` |
| 正文 | `1rem` | `1.5–1.65` |
| 小字 | `0.8125–0.875rem` | `1.45–1.5` |

规则：

- 正文和导航使用 400–500 字重。
- 压缩字体只用于短句和巨大文字。
- 手写字体每个画面最多承担一个词或短语。
- 中文内容不强行模拟拉丁压缩和斜体效果。
- 正文单行保持约 56–68 个字符的阅读宽度。

## 15. 颜色与表面

HTML 界面使用稳定颜色，天空与玻璃颜色由场景光照产生。

| Token | Value | 用途 |
| --- | --- | --- |
| `--ui-black` | `#000000` | 内容过渡、结尾画布 |
| `--ui-ink` | `#17181c` | 浅色表面文字 |
| `--ui-white` | `#ffffff` | 场景上的主文字 |
| `--ui-mist` | `#f5f5f3` | 少量浅色内容表面 |
| `--ui-muted-dark` | `rgba(255,255,255,.68)` | 场景上的次要文字 |
| `--ui-line-dark` | `rgba(255,255,255,.28)` | 深色表面发丝线 |
| `--ui-line-light` | `rgba(0,0,0,.12)` | 浅色表面发丝线 |
| `--ui-focus` | `#9ec5ff` | 键盘焦点 |

- 不为页面再定义传统 light/dark 主题。
- 不使用饱和色填充按钮。
- 不使用默认投影制造层级。
- 不在 HTML 界面叠加大面积彩色渐变模拟天空。

## 16. 间距与形状

- 基础单位：4px。
- 内容最大宽度：1150px。
- 桌面页边距：24–32px。
- 移动端页边距：16–20px。
- 大章节间距：`clamp(80px, 12vw, 160px)`。
- 内容网格间距：24px。

圆角：

| 元素 | 圆角 |
| --- | --- |
| 输入框 | 4px |
| 按钮 | 8px |
| 内容表面 | 12px |
| 媒体 | 11–14px |
| 场景光照控制组 | 12px |

胶囊形状只用于紧凑切换或状态，不作为通用容器。

## 17. 交互与动效

### 动效用途

允许的动效必须至少满足一个目的：

- 表达空间关系
- 解释滚动叙事
- 显示状态变化
- 提供按压反馈
- 防止内容突然跳变

### 时间与缓动

- 普通 UI 动效保持在 300ms 以内。
- 按钮按压：100–160ms，允许 `scale(0.97)`。
- 小型浮层：125–200ms。
- 光照状态插值：600–900ms。
- 首屏滚动场景直接映射滚动进度，不使用固定播放时长。
- 进入和反馈使用 `cubic-bezier(0.23, 1, 0.32, 1)`。
- 屏幕内空间移动使用 `cubic-bezier(0.77, 0, 0.175, 1)`。
- 不使用 `ease-in` 作为界面响应。
- 不使用 `transition: all`。

### 输入方式

- 不劫持滚轮，不强制滚动吸附。
- 键盘导航不增加装饰性延迟。
- Hover 效果只在 `(hover: hover) and (pointer: fine)` 下启用。
- 触摸设备保持 44px 以上目标尺寸。
- 所有动画可以被快速反向或中断，不应从头重新播放。

## 18. 响应式设计

### 桌面

- 展示完整天空、云层深度和玻璃字母骨架。
- 右侧保留垂直光照控制。
- 字形可大幅超出视口边缘。

### 平板

- 降低云层数量和字形横向裁切幅度。
- 保持四套光照和主要滚动变换。
- 导航减少可见栏目。

### 手机

- 重新构图字母骨架，不直接裁切桌面镜头。
- 首帧可以裁切部分字母，但滚动揭示阶段必须完整显示可识别的 `Aqhours`。
- 光照控制移动到右下安全区域。
- 降低像素比、云层层数和玻璃计算成本。
- 首屏滚动距离可以缩短，但叙事顺序不变。
- 不依赖 Hover 或鼠标视差。

## 19. 无障碍

- Canvas 作为装饰层使用，不承载正文和可操作内容。
- 所有文字、导航和按钮使用语义化 HTML。
- 光照按钮提供明确的名称、选中状态和键盘操作。
- 场景上的文字在所有光照状态下保持 WCAG AA 对比度。
- 键盘焦点始终可见。
- 图片和项目媒体提供合适的替代文本。
- 不把重要文字烘焙进场景贴图。

### Reduced motion

当用户启用 `prefers-reduced-motion`：

- 使用当前光照状态的静态场景图。
- 停止字形旋转、视差、云层漂移和滚动空间移动。
- 保留必要的短淡入和颜色反馈。
- 页面内容按正常文档流出现。

## 20. 性能与降级

目标不是在所有设备强制运行同一质量，而是在所有设备保留同一设计意图。

### 渲染等级

| 等级 | 条件 | 行为 |
| --- | --- | --- |
| A | 性能充足的桌面设备 | 完整 WebGL、全部云层、完整玻璃质量 |
| B | 移动设备或中等性能设备 | 降低 DPR、云层数量、反射和采样质量 |
| C | Reduced motion、低性能或 WebGL 不可用 | 使用预渲染静态场景 |

### 目标

- HTML 导航和基础文字不等待 3D 场景才能出现。
- 首屏先显示与当前光照一致的静态 poster，再过渡到实时画布。
- 压缩后的主模型目标不超过约 1.5MB。
- 移动端 poster 目标不超过约 450KB。
- 桌面端 poster 目标不超过约 900KB。
- 场景离开首屏后停止或显著降低渲染频率。
- 桌面目标接近 60fps；移动端最低保持稳定 30fps。
- 不让场景阻塞导航、滚动和首次交互。

- 不在设计文档批准前开始完整页面实现。
